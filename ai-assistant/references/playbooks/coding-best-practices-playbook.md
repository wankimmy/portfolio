# Coding Best Practices Playbook

> If the request is general, ambiguous, or touches many files — ask clarifying yes/no questions **before acting**. Use numbered bullets with explicit answer format: e.g. `1-yes/no  2-A/B`.

Use this when implementing or reviewing code that should be clean, maintainable, and consistent with the existing project.

## Workflow

1. Read the current structure, conventions, and adjacent files first.
2. Identify the best practices that are stack-specific versus universal.
3. Prefer readable code, clear naming, coherent responsibilities, safe validation, and predictable error handling.
4. Keep the change small unless the structure clearly blocks a quality outcome.
5. Decide whether a local improvement is enough or whether the issue should be escalated into code revamp.
6. Check whether the result is easier to understand, safer to modify, and easier to verify than before.

## Output expectation

- repo-consistent best-practice guidance
- implementation quality notes
- local fix versus broader cleanup
- verification considerations
