# UI Delivery Playbook

> If the request is general, ambiguous, or touches many files — ask clarifying yes/no questions **before acting**. Use numbered bullets with explicit answer format: e.g. `1-yes/no  2-A/B`.

Use this when turning designs, screenshots, or verbal UI descriptions into implementation-ready guidance.

## Workflow

1. Identify the core user task on the screen.
2. Break the screen into components, states, and data dependencies.
3. Check responsive behavior and interaction quality.
4. Define loading, error, empty, success, and disabled states.
5. Translate visual structure into implementation primitives.
6. Call out anything ambiguous instead of guessing.

## Output expectation

- screen breakdown
- states and interactions
- responsive behavior
- implementation notes
- open ambiguities
