# Search-First Playbook

> If the request is general, ambiguous, or touches many files — ask clarifying yes/no questions **before acting**. Use numbered bullets with explicit answer format: e.g. `1-yes/no  2-A/B`.

Use this when deciding whether to adopt an existing solution or build custom behavior.

## Workflow

1. Define the capability, stack, constraints, and acceptance criteria.
2. Search the current repo for existing code, scripts, skills, or conventions that already solve part of the problem.
3. Check the current framework, tool surface, and local workspace capabilities before adding new dependencies.
4. Compare the best options using adopt, extend, wrap, or build.
5. Recommend the smallest path that meets the need with the lowest long-term cost.
6. If choosing custom implementation, explain why the alternatives were not good enough.

## Output expectation

- capability and constraints
- repo-local options checked
- top candidate approaches
- recommendation with tradeoffs
- risks and limitations
- smallest next step
