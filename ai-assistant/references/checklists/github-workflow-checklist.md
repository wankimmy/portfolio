# GitHub Workflow Checklist

> If the request is general, ambiguous, or touches many files, ask clarifying yes/no questions before acting.

- Is the GitHub object identified: issue, PR, Actions run, release, Dependabot alert, repo setting, or project board?
- Is remote state confirmed through GitHub MCP, `gh`, or user-provided evidence before making claims?
- Does PR or issue content include context, scope, risks, verification, rollback, and linked work?
- Are CI failures traced to the first root failing step, not only the final cascade?
- Are labels, milestones, comments, reruns, merges, or settings changes requested or clearly authorized?
- Are secrets, private URLs, and sensitive logs kept out of generated comments?
- Is the recommended next action tied to owner, risk, and verification?
