# Learning Promotion Checklist

> If the request is general, ambiguous, or touches many files — ask clarifying yes/no questions **before acting**. Use numbered bullets with explicit answer format: e.g. `1-yes/no  2-A/B`.

Use this after meaningful tasks, reviews, or decisions to decide where a lesson should live.

## Put it in memory when:

- it is a durable fact
- it is useful future context
- it is not yet repeated enough to become a stronger rule

## Promote it to a checklist when:

- it is a repeatable verification step
- forgetting it would likely cause regressions

## Promote it to a pitfall when:

- it is a recurring trap
- it has caused defects, confusion, or wasted effort more than once

## Promote it to a playbook when:

- it is a reusable workflow
- it explains how to execute a common task safely

## Promote it to a skill when:

- it changes how the assistant should repeatedly reason or work
- it improves performance across many future sessions

## Automatic promotion rule

- If repeated usage reveals a missing reusable capability, do not leave it only in memory.
- Create or update the strongest fitting artifact:
  - skill
  - checklist
  - playbook
  - pitfall
  - rule
- Promote only durable, broadly reusable improvements, not one-off hacks.

## Quality bar

- keep notes brief
- record the source signal and confidence when the lesson is not self-evident
- record the promotion target or defer reason so the learning can be audited later
- prefer specific triggers over vague reminders
- avoid duplicate learnings in multiple places unless one points to the stronger artifact
