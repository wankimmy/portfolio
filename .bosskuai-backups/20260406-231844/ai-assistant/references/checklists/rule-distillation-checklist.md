# Rule Distillation Checklist

> If the request is general, ambiguous, or touches many files — ask clarifying yes/no questions **before acting**. Use numbered bullets with explicit answer format: e.g. `1-yes/no  2-A/B`.

- Does the candidate principle appear in more than one skill, reference, or repeated workflow?
- Would promoting it to a rule clearly change future assistant behavior?
- Is the principle broad enough to be a rule rather than a project-specific tactic or example?
- Is the same idea already covered in `AGENTS.md`, `CLAUDE.md`, `.claude/rules/`, `.cursor/rules/`, or `.codex/AGENTS.md`?
- Is the rule wording actionable, concise, and durable rather than tied to one temporary implementation detail?
- Would adding this rule reduce fragmentation, or would it only duplicate existing guidance?
- If revising an existing rule, is the current wording actually insufficient or inaccurate?
- Did you avoid treating rule changes as automatic and preserve review before application?
